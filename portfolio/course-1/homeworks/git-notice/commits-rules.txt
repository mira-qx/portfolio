
Правила оформления сообщений коммитов по Conventional Commits.

Сообщение коммита должно иметь следующую структуру:

<тип>[(область)]: <описание>

[необязательное тело]

[необязательный нижний колонтитул]

Основные правила и элементы:

<тип> (обязательно): Определяет категорию изменений. Это слово, которое должно быть одним из предопределенных (например, feat, fix, chore).
(область) (необязательно): Указывает часть проекта, к которой относится коммит (например, название компонента, файла или модуля). Заключается в круглые скобки сразу после типа.

: (обязательно): Пробел после двоеточия.
<описание> (обязательно): Краткое описание изменений. Рекомендуется использовать не более 50 символов в заголовке, писать его на английском языке, с маленькой буквы и без точки в конце. Описание должно начинаться с глагола в повелительном наклонении.

[необязательное тело]: Более подробная информация об изменениях. Отделяется от заголовка пустой строкой. Каждая строка тела не должна превышать 72 символа.

[необязательный нижний колонтитул] (footer): Используется для указания метаинформации, например, ссылок на задачи в трекере или информации о критических изменениях. 

Примеры:

feat(authentication): add a login button (Добавление новой функциональности в модуль аутентификации)
fix: correct a typo in the main page (Исправление ошибки)
chore(deps): update dependency lodash to v4 (Рутинная задача, не связанная с кодом)
docs: update README with new instructions (Изменение документации)
BREAKING CHANGE: remove support for Node.js 10 (Критическое изменение, требующее обновления мажорной версии)

Типы коммитов:

feat - Новая функция	
fix - Исправление ошибки	
docs - Изменения в документации	
style - Изменения, не влияющие на логику 
refactor - Рефакторинг кода (без исправления ошибок и добавления новых функций)	
perf - Изменения, улучшающие производительность	
test - Добавление или исправление тестов	
chore- Рутинные задачи, не касающиеся основного кода 
build - Изменения, влияющие на систему сборки или внешние зависимости	
ci - Изменения в CI/CD	

Критические изменения (BREAKING CHANGE):

Любой коммит, который вносит критические изменения (т.е. изменения, несовместимые с предыдущими версиями API), должен содержать текст BREAKING CHANGE: в теле или нижнем колонтитуле сообщения. Также можно добавить восклицательный знак ! после типа или области, чтобы явно указать на критическое изменение, например: feat!: introduce new API. Это автоматически приводит к обновлению MAJOR (мажорной) версии проекта.
